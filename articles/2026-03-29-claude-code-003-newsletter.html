1. チャットボットとプラットフォームの決定的な違い 1. The Decisive Difference Between Chatbots and Platforms

ChatGPT、Gemini、Claude.ai — ブラウザ型 AI は「問いを投げて答えを受け取る」ツールだ。強力だが、構造的な限界がある。セッションが終われば文脈は消え、次の会話はゼロから始まる。ユーザーの業務プロセスに組み込まれることはなく、あくまで「相談相手」に留まる。 ChatGPT, Gemini, Claude.ai — browser-based AI tools are built around a simple loop: ask a question, receive an answer. Powerful, yes, but structurally limited. When a session ends, context evaporates. The next conversation starts from scratch. These tools are never embedded in a user's business processes; they remain advisors at best.

Claude Code が根本的に異なるのは、ユーザーが自分の業務ロジックをシステムに組み込める点にある。4つの拡張レイヤー — Agents、Skills、Hooks、MCP — がその仕組みだ。 Claude Code is fundamentally different because users can embed their own business logic into the system. Four extensibility layers — Agents, Skills, Hooks, and MCP — make this possible.

核心: チャットボットは「あなたの質問」を待つ。プラットフォームは「あなたの設計」を待つ。Claude Code は後者である。 Core insight: A chatbot waits for your questions. A platform waits for your design. Claude Code is the latter.

2. Agents — 独立した文脈を持つ自律ワーカー 2. Agents — Autonomous Workers with Isolated Context

Agent（サブエージェント）とは、メインの会話スレッドから独立したコンテキストウィンドウで動作する、特化型の AI ワーカーである。メインスレッドが「プロジェクトマネージャー」なら、Agent は「専門家チーム」に相当する。 An Agent (sub-agent) is a specialised AI worker that operates in its own context window, independent of the main conversation thread. If the main thread is the "project manager," Agents are the "specialist team."

設計原則は3つ: Three design principles govern Agents:

  1. コンテキスト分離 — Agent は親スレッドとメモリを共有しない。メインの会話が大量のリサーチ結果で汚染されることを防ぐ Context isolation — Agents do not share memory with the parent thread, preventing the main conversation from being polluted by large volumes of research output
  2. ツール権限の制御 — Agent ごとに使用可能なツールを制限できる。「リサーチ用 Agent はファイル編集不可」といった最小権限設計が可能 Tool permission control — Available tools can be restricted per Agent. "Research Agent cannot edit files" — this kind of least-privilege design is built in
  3. 並列実行 — 独立した Agent は同時に起動できる。Claude Code は最大7つの同時操作をサポートしており、セキュリティレビュー・パフォーマンス分析・型チェックを並列で走らせることで直列実行の数倍の速度が得られる Parallel execution — Independent Agents can run concurrently. Claude Code supports up to 7 simultaneous operations, enabling security reviews, performance analyses, and type checks to run in parallel at several times the speed of sequential execution

Agent は Markdown ファイル + YAML フロントマターで定義する。プログラミング言語は不要だ: Agents are defined using Markdown files with YAML front matter. No programming language is required:

# .claude/agents/code-reviewer.md --- name: code-reviewer description: Review code quality, security, and maintainability tools: Read, Grep, Glob, Bash model: sonnet maxTurns: 50 --- You are a code review specialist...

主要フィールド: name（識別子）、tools（許可ツール）、model（使用モデル）、maxTurns（最大ターン）、memory（永続メモリ）、isolation（worktree 隔離）、mcpServers（専用 MCP サーバー）。 Key fields: name (identifier), tools (permitted tools), model (model selection), maxTurns (maximum turns), memory (persistent memory), isolation (worktree isolation), mcpServers (dedicated MCP servers).

注意: Agent は子 Agent を生成できない（無限再帰防止）。複雑なワークフローはメインスレッドが複数 Agent を並列起動するオーケストレーションパターンで実現する。 Note: Agents cannot spawn child Agents (to prevent infinite recursion). Complex workflows are achieved via an orchestration pattern where the main thread launches multiple Agents in parallel.

Agent Teams — 2026年2月の新機能 Agent Teams — New Feature from February 2026

2026年2月5日、Anthropic は Opus 4.6 とともに Agent Teams をリサーチプレビューとしてリリースした。従来のサブエージェントが「上司に報告する部下」だったのに対し、Agent Teams は「互いに連携する協働チーム」だ。 On 5 February 2026, Anthropic released Agent Teams as a research preview alongside Opus 4.6. Where traditional sub-agents were "subordinates reporting to a manager," Agent Teams are "collaborative peers coordinating with one another."

象徴的な事例: Nicholas Carlini は16のエージェントに Rust ベースの C コンパイラをゼロから書かせ、約2,000セッション・$20,000 の API コストで、Linux 6.9 を x86・ARM・RISC-V でビルドできる10万行のコンパイラを生成した。 A landmark case: Nicholas Carlini directed 16 agents to write a Rust-based C compiler from scratch. Over roughly 2,000 sessions at $20,000 in API costs, they produced a 100,000-line compiler capable of building Linux 6.9 for x86, ARM, and RISC-V.

3. Skills — 再利用可能なプロンプト・パッケージ 3. Skills — Reusable Prompt Packages

Skill はスラッシュコマンド（/deploy、/review 等）として呼び出せる、再利用可能なプロンプト指示書だ。「毎回同じ指示を書く」非効率を排除し、ベストプラクティスをパッケージ化する。 Skills are reusable prompt instruction sets invoked as slash commands (/deploy, /review, etc.). They eliminate the inefficiency of writing the same instructions every time, packaging best practices into callable units.

Agent が「誰がやるか」を定義するなら、Skill は「何をやるか」を定義する。 If Agents define "who does it," Skills define "what gets done."

~/.claude/skills/deploy/ ├── SKILL.md # Required: front matter + instructions ├── checklist.md # Optional: supplementary material └── scripts/ └── validate.sh # Optional: helper scripts

注目すべき機能: Notable features:

  • context: fork — Skill を独立した Agent コンテキストで実行。メインの会話を汚さずに複雑な作業を遂行 context: fork — Execute the Skill in an isolated Agent context, performing complex work without polluting the main conversation
  • 動的コンテキスト注入 — !バッククォート構文でシェルコマンドの結果をプロンプトに動的注入。!`gh pr diff` は Claude の処理開始前にシェルで実行され、出力がプロンプトに展開される Dynamic context injection — Use !backtick syntax to dynamically inject shell command output into prompts. !`gh pr diff` executes in the shell before Claude begins processing, and the output is expanded into the prompt
  • Commands との統合 — 2026年現在、.claude/commands/*.md と .claude/skills/*/SKILL.md は統合されており、どちらも同じ /slash-command インターフェースを作る Commands integration — As of 2026, .claude/commands/*.md and .claude/skills/*/SKILL.md are unified, both producing the same /slash-command interface

Skill + Agent の接続: context: fork + agent: で両者を接続すると、「この専門家にこの仕事を頼む」というパターンが成立する。 Skill + Agent connection: Combining context: fork with agent: creates a "delegate this task to that specialist" pattern.

4. Hooks — 確率ではなくコードで強制する 4. Hooks — Enforced by Code, Not Probability

Hooks は Claude Code のライフサイクルイベントに応答して自動実行されるシェルコマンドだ。Agent や Skill が「AI の判断」に依存するのに対し、Hooks は決定論的に動作する。 Hooks are shell commands that automatically execute in response to Claude Code lifecycle events. Whilst Agents and Skills depend on "AI judgement," Hooks operate deterministically.

これが重要な理由: AI は確率的に動作する。「絶対にやってはいけないこと」をプロンプトで禁止しても、100%の遵守は保証できない。Hooks は確率ではなく、コードで強制する。 Why this matters: AI operates probabilistically. Even if you forbid something via prompt, 100% compliance cannot be guaranteed. Hooks enforce rules through code, not probability.

Hook の戻り値: Exit code 0 はアクション続行、Exit code 2 はブロック。ブロック理由を stderr に書くと Claude がそれを読んで代替手段を考える。 Hook return values: exit code 0 continues the action; exit code 2 blocks it. Writing the reason to stderr allows Claude to read it and consider alternatives.

Prompt Hooks という新しい種別もある — LLM が別の LLM のアクションをリアルタイムで検証する仕組みで、1チェックあたり数セントのコストで動作する。 A newer variant called Prompt Hooks also exists — a mechanism where one LLM verifies another LLM's actions in real-time, operating at a cost of a few cents per check.

設計原則: Hooks の量産は解決策ではない。根本対策は Skill の手順に正しいワークフローを組み込むこと。Hooks はバックストップとして最小限に留める。 Design principle: Mass-producing Hooks is not the solution. The root fix is embedding the correct workflow into Skill procedures. Hooks should remain minimal as a backstop.

5. MCP — 外部世界への接続プロトコル、業界標準へ 5. MCP — The Connection Protocol Going Industry-Standard

Model Context Protocol (MCP) は Claude Code を外部ツール・API・データベースと接続するオープンソース標準プロトコルだ。 The Model Context Protocol (MCP) is an open-source standard protocol that connects Claude Code to external tools, APIs, and databases.

MCP サーバーは Agent にスコープすることも可能。「データベース分析 Agent だけが PostgreSQL に接続できる」という最小権限設計を実現する。 MCP servers can be scoped to specific Agents, enabling a least-privilege design where "only the database analysis Agent can connect to PostgreSQL."

6. 4つのレイヤーの組み合わせパターン 6. Composition Patterns Across the Four Layers

各コンポーネントは単独でも有用だが、真価は組み合わせにある。 Each component is useful on its own, but the real power lies in composition.

Pattern A: Skill + Agent（タスク委譲） — /code-review Skill が context: fork で専用 Agent にレビューを委譲。メインスレッドはレビュー結果だけを受け取り、コンテキストを消費しない。 Pattern A: Skill + Agent (task delegation) — A /code-review Skill delegates to a dedicated Agent via context: fork. The main thread receives only the review results without consuming context.

Pattern B: Hook + Skill（自動品質保証） — PostToolUse Hook がファイル編集を検知し自動で Prettier を実行。Skill にもワークフローを記述し、Hook は最終防衛線として機能。 Pattern B: Hook + Skill (automated quality assurance) — A PostToolUse Hook detects file edits and automatically runs Prettier. The Skill also documents the workflow; the Hook serves as the last line of defence.

Pattern C: Agent + MCP（スコープ付き外部接続） — データ分析 Agent に PostgreSQL の MCP サーバーのみを付与。ファイル編集や GitHub 操作は不可。最小権限の原則がアーキテクチャレベルで実現。 Pattern C: Agent + MCP (scoped external connection) — Grant a data analysis Agent access only to a PostgreSQL MCP server. No file editing or GitHub operations permitted. Least-privilege at the architectural level.

Pattern D: 並列 Agent オーケストレーション — メインスレッドが3つの Agent を並列起動（セキュリティレビュー・パフォーマンス分析・テスト実行）。直列の3倍以上の速度で、メインのコンテキストを汚さない。 Pattern D: Parallel Agent orchestration — The main thread launches three Agents in parallel (security review, performance analysis, test execution). Runs at over 3x the speed of sequential processing without polluting the main context.

設計の指針: 「何を」= Skill、「誰が」= Agent、「強制」= Hook、「外部接続」= MCP。この4軸でワークフローを分解すれば最適な構成が見える。 Design heuristic: "What" = Skill, "Who" = Agent, "Enforce" = Hook, "Connect" = MCP. Decompose your workflow along these four axes to find the optimal configuration.

7. Plugins とエコシステムの現在地 7. Plugins and the Current Ecosystem

Skills・Agents・Hooks・MCP サーバーの定義を一つにパッケージ化し配布する仕組みが Plugins だ。 Plugins are the mechanism for packaging and distributing Skills, Agents, Hooks, and MCP server definitions as a single unit.

my-plugin/ ├── plugin.json # Metadata ├── skills/ # Skill definitions ├── agents/ # Agent definitions ├── hooks/ # Hook configuration └── .mcp.json # MCP server definitions

配布は GitHub リポジトリ経由で claude plugin install コマンドでインストール。Anthropic 公式のプラグインレジストリ anthropics/claude-plugins-official も GitHub 上に存在する。コミュニティによるキュレーションリスト awesome-claude-code も活発に更新されている。 Distribution occurs via GitHub repositories using the claude plugin install command. Anthropic maintains an official plugin registry at anthropics/claude-plugins-official on GitHub, and the community-curated awesome-claude-code list is actively updated.

エンタープライズ環境では、マネージド設定で組織全体にプラグインを配布可能。「全社で同じコードレビュー基準」「承認済み MCP サーバーのみ許可」といったガバナンスが実現される。 In enterprise environments, managed settings enable organisation-wide plugin distribution. Governance such as "uniform code review standards company-wide" or "only approved MCP servers permitted" becomes achievable.
