概要Summary

Claude Codeの潜在能力を引き出すには4つの習慣が必要だ——①プロンプトに「役割・制約・フォーマット・例示」を組み込む、②コンテキストウィンドウの消費を意識してリセットタイミングを管理する、③案件・業務別にCLAUDE.mdを構築してAIに文脈を記憶させる、④よくある失敗パターンを知って回避する。これを実践すれば、Claude Codeは「たまに使うツール」から「毎日の生産性エンジン」になる。 Four habits unlock Claude Code's potential: ① embed Role, Constraints, Format, and Examples in every prompt; ② monitor context window consumption and manage reset timing; ③ build context-rich CLAUDE.md files per engagement; ④ know and avoid the most common failure patterns. Master these and Claude Code transforms from an occasional tool into a daily productivity engine.

Explicitly specifying Role, Constraints, Format, and Examples dramatically shifts output quality.   • Q2 コンテキストウィンドウとは何か、なぜ管理が必要か？ — Claude Codeは200kトークンの文脈を持つが、長い会話・大量ファイル読み込みで消耗すると後半の回答品質が落ちる。「新しい作業は新しいセッション」という原則と、/compact・/clearコマンドの使い方を理解することが品質維持の鍵だ。 What is the context window and why does it need managing? — Claude Code holds 200k tokens of context, but quality degrades as long conversations and heavy file loading consume it. 1. プロンプト設計の4原則 1. The 4 Principles of Prompt Design

Claude Codeに「〇〇して」と指示するだけでも動く。しかし出力の品質は、指示の構造によって劇的に変わる。Finance Proが覚えるべき4要素は次のとおりだ。 Claude Code will respond to a simple "do X" instruction. But output quality shifts dramatically based on how you structure the instruction. The four elements Finance Pros need to internalise are as follows.

  • 役割 (Role): AIにどの専門家として振る舞わせるか。「M&AアドバイザーのシニアアナリストAIアシスタント」と定義するだけで、回答のトーン・粒度・前提知識が変わる。Role: Which expert persona should the AI adopt. Defining it as "a senior analyst AI assistant at an M&A advisory firm" alone changes tone, granularity, and assumed domain knowledge.
  • 制約 (Constraints): 会計基準・通貨・出力言語・精度要件。「IFRS、百万円単位、不確かな情報には[要確認]タグ」のような具体的な制約が回答の信頼性を高める。Constraints: Accounting standard, currency, output language, accuracy requirements. Specific constraints like "IFRS, in millions, flag uncertain information with [verify]" raise output reliability.
  • フォーマット (Format): 表・箇条書き・レポート・JSON・HTML。出力形式を明示することが最も効果の大きい改善だ。Excelに貼り付けられる表が欲しいならその列構成を書く。HTMLレポートが欲しいならそのまま言う。Format: Table, bullet list, report, JSON, HTML. Specifying output format is the single highest-impact improvement. Want a table you can paste into Excel — write the column structure. Want an HTML report — say so.
  • 例示 (Examples): few-shotで期待するアウトプット構造を見せる。「以下のサンプルと同じ構造で」と1つの例を渡すだけで、AIは意図を正確に読み取る。Examples: Show the expected output structure via few-shot examples. Providing one sample with "follow this structure" precisely communicates intent in a way words alone cannot.

この4要素を「すべて同時に書く」必要はない。しかし意識して取捨選択することで、最初の一発で意図した出力が得られる確率が大きく上がる。以下に典型的な悪例と良例を示す。 You don't need all four elements every time. But consciously selecting which to include dramatically increases the probability of getting the intended output on the first attempt. Below are representative bad and good examples.

4原則の中で最も効果が大きいのはフォーマット指定だ。Excelに貼り付けられる表が欲しいならその構造を書く。HTMLレポートが欲しいならそのまま言う。AIは指示された出力形式に従う——曖昧にすると、AIが「適切だと判断した」形式で出力する。「また形式が違う」という繰り返しのやり取りは、フォーマット指定一行で消える。 Format specification has the highest impact of the four principles. If you want a table you can paste into Excel, specify that structure. If you want an HTML report, say so. AI follows the output format it is given — leave it ambiguous and you get whatever format the AI judges appropriate. The repetitive back-and-forth of "wrong format again" disappears with a single format line.

2. コンテキストウィンドウを理解する 2. Understanding the Context Window

Claude Codeのコンテキストウィンドウは200kトークン——日本語の長編小説1冊分に相当する広大な「作業記憶」だ。しかしFinance Proの実務では、これを想像以上に早く消費する。 Claude Code's context window holds 200k tokens — roughly equivalent to a full-length novel. But in Finance Professional workflows, you consume it faster than you might expect.

有価証券報告書100p PDF100-page securities filing (PDF) ~40,000 20%
月次P&L Excel（3シート）Monthly P&L Excel (3 sheets) ~8,000 4%
長い会話30往復30-turn conversation ~15,000 8%
コード実行出力（1回）Code execution output (per run) ~2,000 1%
DDレポート5本分5 DD reports ~100,000 50%

問題はトークン数そのものではなく、コンテキスト汚染だ。会話が長くなるにつれ、初期の指示（「IFRSで解釈せよ」「百万円単位で」）が後半になるほど薄く扱われ始める。AIの「注意」は均等ではなく、直近の内容に引っ張られる。 The issue isn't token count per se — it's context pollution. As conversations grow longer, early instructions ("interpret under IFRS," "in millions") receive progressively less weight. AI attention is not uniform; it's pulled toward recent content.

2つのリセットコマンド Two Reset Commands

TERMINAL — コンテキスト管理コマンド
/compact   # 会話を要約してコンテキストを圧縮する（作業継続したい場合）
/clear     # コンテキストを完全にリセット（新規セッション相当）

/compactは長くなった会話を要約してトークンを節約しながら作業を継続する。/clearは完全に白紙に戻す——新しい案件・新しいタスクの開始時に使う。 /compact summarises the conversation to save tokens while continuing work. /clear resets to a blank slate — use it when starting a new engagement or a distinct new task.

「1つの作業タスク = 1セッション」——これがコンテキスト管理の基本原則だ。 "One task, one session" — this is the fundamental principle of context management.

DDでVDRの50ファイルを一気に読み込もうとするのは悪手だ。10ファイルずつバッチに分けてセッションを分けると、品質が安定する。「全部一度に」はコンテキスト汚染の最短経路だ。 Attempting to load 50 VDR files in a single session is counterproductive. Batch them into groups of 10 per session and quality stabilises. "Everything at once" is the fastest route to context pollution.

3. CLAUDE.mdの本格構築 3. Building a Production CLAUDE.md

CLAUDE.mdはClaude Codeが起動時に必ず読み込む設定ファイルだ。「AIへのオンボーディング文書」として書くとよい——新人スタッフに渡す業務説明書と同じ感覚で。推奨するのは2層構造だ。 CLAUDE.md is the configuration file Claude Code reads at every startup. Think of it as an AI onboarding document — written the same way you would brief a new team member. The recommended structure is two layers.

  • グローバルCLAUDE.md（~/.claude/CLAUDE.md）: すべての作業に適用される普遍的設定。言語・通貨・会計基準・自分のバックグラウンドを書く。Global CLAUDE.md (~/.claude/CLAUDE.md): Universal settings applied to all work. Language, currency, accounting standard, your professional background.
  • プロジェクトCLAUDE.md（./CLAUDE.md）: 案件・業務フォルダに置く案件固有設定。対象会社・基準日・フォルダ構造・禁止事項を書く。Project CLAUDE.md (./CLAUDE.md): Engagement-specific settings placed in each deal/work folder. Target company, reference date, folder structure, prohibited actions.
グローバルCLAUDE.md（~/.claude/CLAUDE.md）

Finance Pro の視点 — チーム共有の威力Finance Pro Perspective — The Power of Team Sharing

CLAUDE.mdの最大の価値は「チーム共有」だ。案件フォルダにCLAUDE.mdを置いてgit管理すれば、チームメンバー全員が同じコンテキストでClaudeを使える。マネージャーがCLAUDE.mdを書き、スタッフがそのコンテキストでClaudeに作業させる——これが次世代のチームワークフローだ。「Claudeに何を教えたか」の差が、チームの生産性差になる。 CLAUDE.md's greatest value is team-sharing. Place it in the deal folder under git version control and every team member uses Claude with identical context. The manager writes the CLAUDE.md; staff members direct Claude within that context — this is the next-generation team workflow. The gap in "what each team told Claude" becomes the productivity gap between teams.

4. よくある失敗パターンと回避策 4. Common Failure Patterns and Fixes

Claude Codeを使い始めた Finance Pro が頻繁に踏む落とし穴を整理する。知っておくだけで時間のロスが大きく減る。 Here are the pitfalls Finance Pros most commonly encounter when starting with Claude Code. Knowing them in advance saves significant wasted time.

① 「なんでもやってください」問題 — 指示が広すぎる ① "Do everything" problem — instruction too broad

症状: 「このフォルダを分析してください」と言ったら、期待と全然違う出力が返ってきた。
回避策: タスクを1つに絞り、「このファイルの、このデータの、この形式で」まで具体化する。複数タスクは複数の指示に分割する。 Symptom: You asked "analyse this folder" and got output completely different from what you needed.
Fix: Narrow to one task and specify "this file, this data, in this format." Split multiple tasks into separate instructions.

② コンテキスト引き継ぎミス — 「さっき言ったこと」が通じない ② Context drift — "I told you earlier" stops working

症状: 長い会話の後半で、冒頭に伝えた制約（IFRS・百万円単位）が無視されている。
回避策: /compactでリセット後、新しい会話冒頭で制約を再掲する。恒久的な制約はCLAUDE.mdに書くことで根本解決できる。 Symptom: Deep into a long conversation, the constraints you stated at the start (IFRS, in millions) are being ignored.
Fix: After /compact, restate constraints at the top of the new conversation. Permanently solving this: write standing constraints into CLAUDE.md.

③ ファイルパスエラー — 「ファイルが見つかりません」 ③ File path errors — "file not found"

症状: ファイル名を指定したのに「存在しない」と言われる。
回避策: まず「今どのディレクトリにいるか確認して」と聞く。フルパスまたは相対パスを正確に渡す。WSL2使用時はWindowsのバックスラッシュに注意（\ → /）。 Symptom: You specified a filename but Claude says it doesn't exist.
Fix: First ask Claude to confirm the current directory. Pass the full or accurate relative path. When using WSL2, watch for Windows backslash issues (\ → /).

④ 「確認なしに書き込んだ」問題 — 元データが上書きされた ④ Silent writes — source data overwritten without warning

症状: 分析中に原本ファイルが変更・上書きされた。
回避策: CLAUDE.mdの禁止事項に「input/フォルダへの書き込み禁止」を明記。重要な作業前にgit commitでスナップショットを保存する習慣をつける。 Symptom: Source files were modified or overwritten during analysis.
Fix: Explicitly prohibit "no writes to the input/ folder" in your CLAUDE.md. Make a habit of running git commit before significant operations to create a recoverable snapshot.

⑤ 数値の「それっぽい嘘」 — ハルシネーション ⑤ Plausible-looking errors — hallucination

症状: 計算結果が一見正しそうに見えるが、元データと照合すると誤っている。
回避策: 「計算根拠（数式・参照元）を必ず示せ」をCLAUDE.mdに書く。財務数値は必ず元データと照合する。「AIが言ったから正しい」は禁物——特に重要な意思決定に使う数値は必ずダブルチェックをルール化する。 Symptom: A calculation result looks correct but doesn't match the source data on verification.
Fix: Add "always show calculation basis (formula and source)" to your CLAUDE.md. Always verify financial figures against source data. Never treat "AI said so" as sufficient — especially for figures that inform significant decisions, double-checking must be a non-negotiable rule.

5. Finance Pro の日常的な使い方パターン 5. Daily Usage Patterns for Finance Pros

ここまでの4原則を日常業務に落とし込んだ具体例を示す。月次業績レポートの自動生成は、Finance Proが最初に自動化すべき定番ルーティンだ。 Here is a concrete application of the four principles to daily work. Monthly performance report generation is the canonical routine Finance Pros should automate first.

CLAUDE CODE — 月次業績レポート自動生成プロンプト（テンプレート）

【制約】

1. 売上高・粗利率・営業利益率のサマリー表を作成
2. 前月比・前年同月比・予実差異を各指標に付記
3. 異常値（前月比±20%超）があればコメントを添付

【フォーマット】
HTMLレポートとして output/report_2026_03.html に保存。
経営会議提出用レイアウト（A4印刷対応）。

このプロンプトを月次でコピペして月と年を変えるだけで、毎月のレポート生成が半自動化される。最初の1時間をプロンプト設計に投資することで、毎月2〜3時間の作業が5分に変わる。 Copy this prompt monthly, update the date, and report generation becomes semi-automated. Investing one hour in prompt design upfront converts 2–3 hours of monthly work into 5 minutes.

Finance Pro の視点 — プロンプトも「資産」として管理するFinance Pro Perspective — Treat Prompts as Assets

設計したプロンプトはprompts/フォルダにMarkdownファイルとして保存してgit管理する。「月次レポート用」「DDサマリー用」「開示チェック用」と業務別に整理すれば、チームで共有・改善できるプロンプトライブラリになる。AIへの指示は、いまや業務ノウハウそのものだ。 Save your designed prompts as Markdown files in a prompts/ folder under git version control. Organise by task — "monthly report," "DD summary," "disclosure check" — and you have a team-shareable, continuously improving prompt library. Instructions to AI are now institutional knowledge in their own right.
