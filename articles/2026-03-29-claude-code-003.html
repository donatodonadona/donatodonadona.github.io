1. チャットボットとプラットフォームの決定的な違い1. The Decisive Difference Between a Chatbot and a Platform

ChatGPT、Gemini、Claude.ai — ブラウザ型 AI は「問いを投げて答えを受け取る」ツールである。強力だが、構造的な限界がある。セッションが終われば文脈は消え、次の会話はゼロから始まる。ユーザーの業務プロセスに組み込まれることはなく、あくまで「相談相手」に留まる。 ChatGPT, Gemini, Claude.ai — browser-based AI tools are designed around "ask a question, get an answer." They are powerful, but structurally limited. When a session ends, context evaporates and the next conversation starts from zero. They are never embedded in your work process — they remain a "consultant" you visit.

Claude Code が根本的に異なるのは、ユーザーが自分の業務ロジックをシステムに組み込める点にある。4つの拡張レイヤー — Agents、Skills、Hooks、MCP — がその仕組みだ。 Where Claude Code fundamentally differs is that users can embed their own business logic into the system. Four extensibility layers — Agents, Skills, Hooks, and MCP — provide the mechanism.

Session (メイン会話スレッド) | |-- Skills 再利用可能なプロンプト指示書 | |-- 一部は Agents をフォークして実行 | |-- Agents 独立コンテキストで動く自律ワーカー | |-- 独自ツール権限 | |-- 独自メモリ | |-- 独自 MCP サーバー | |-- Hooks ライフサイクルイベントへの自動応答 | |-- PreToolUse (実行前バリデーション) | |-- PostToolUse (実行後フォーマット) | |-- Stop (最終検証) | |-- MCP 外部ツール・API・DB の統合プロトコル |-- stdio (ローカルプロセス) |-- HTTP (リモートサーバー)Session (Main conversation thread) | |-- Skills Reusable prompt instruction packages | |-- Some fork into Agents for execution | |-- Agents Autonomous workers with isolated context | |-- Own tool permissions | |-- Own persistent memory | |-- Own MCP servers | |-- Hooks Automatic responses to lifecycle events | |-- PreToolUse (pre-execution validation) | |-- PostToolUse (post-execution formatting) | |-- Stop (final verification) | |-- MCP Integration protocol for external tools/APIs/DBs |-- stdio (local processes) |-- HTTP (remote servers)

核心: チャットボットは「あなたの質問」を待つ。プラットフォームは「あなたの設計」を待つ。Claude Code は後者である。 Core insight: A chatbot waits for your questions. A platform waits for your design. Claude Code is the latter.

2. Agents — 独立した文脈を持つ自律ワーカー2. Agents — Autonomous Workers with Isolated Context

2.1 Agents とは何か2.1 What are Agents?

Agent（サブエージェント）とは、メインの会話スレッドから独立したコンテキストウィンドウで動作する、特化型の AI ワーカーである。メインスレッドが「プロジェクトマネージャー」なら、Agent は「専門家チーム」に相当する。 Agents (subagents) are specialised AI workers that operate in their own independent context window, separate from the main conversation thread. If the main thread is the "project manager," Agents are the "specialist team."

重要な設計原則は3つある: Three design principles are critical:

  • 1 コンテキスト分離 — Agent は親スレッドとメモリを共有しない。これにより、メインの会話が大量のリサーチ結果やログで汚染されることを防ぐ Context isolation — Agents do not share memory with the parent thread. This prevents the main conversation from being polluted by large volumes of research results or logs
  • 2 ツール権限の制御 — Agent ごとに使用できるツールを制限できる。「リサーチ用 Agent はファイル編集不可」「レビュー用 Agent は Bash 不可」といった設計が可能 Tool permission control — Available tools can be restricted per Agent. Designs like "research Agent cannot edit files" or "review Agent cannot use Bash" become possible
  • 3 並列実行 — 独立した Agent は同時に起動できる。セキュリティレビュー、パフォーマンス分析、型チェックを並列で走らせることで、直列実行の数倍の速度で結果が得られる Parallel execution — Independent Agents can run concurrently. Running security review, performance analysis, and type checking in parallel yields results several times faster than sequential execution

2.2 Agent の定義方法2.2 How to define an Agent

Agent は Markdown ファイル + YAML フロントマターで定義する。プログラミング言語は不要だ。 Agents are defined as Markdown files with YAML frontmatter. No programming language is needed.

# ファイル: .claude/agents/code-reviewer.md --- name: code-reviewer description: コード変更後に品質・セキュリティ・保守性をレビュー tools: Read, Grep, Glob, Bash model: sonnet maxTurns: 50 --- あなたはコードレビューの専門家です。 変更されたファイルを分析し、以下の観点で問題を報告してください: - セキュリティ脆弱性（CRITICAL） - パフォーマンス問題（HIGH） - 可読性・保守性（MEDIUM）# File: .claude/agents/code-reviewer.md --- name: code-reviewer description: Reviews quality, security, and maintainability after code changes tools: Read, Grep, Glob, Bash model: sonnet maxTurns: 50 --- You are a code review specialist. Analyse the changed files and report issues from these perspectives: - Security vulnerabilities (CRITICAL) - Performance issues (HIGH) - Readability and maintainability (MEDIUM)

主要なフロントマターフィールドを整理する: The key frontmatter fields are:

フィールドField 役割Purpose 例Example
name 一意の識別子Unique identifier code-reviewer
tools 使用許可ツールのホワイトリストAllowlist of permitted tools Read, Grep, Glob
model 使用モデルの指定Model override sonnet / opus / haiku
maxTurns 最大ターン数Max agentic turns 50
memory 永続メモリのスコープPersistent memory scope project / user / local
isolation Git worktree での隔離実行Run in isolated git worktree worktree
mcpServers この Agent 専用の MCP サーバーMCP servers scoped to this Agent postgres, github
注意Caveat

Agent は子 Agent を生成できない（無限再帰防止）。複雑なワークフローは、メインスレッドが複数の Agent を並列で起動するオーケストレーションパターンで実現する。 Agents cannot spawn child Agents (to prevent infinite recursion). Complex workflows are achieved through an orchestration pattern where the main thread launches multiple Agents in parallel.

3. Skills — 再利用可能なプロンプト・パッケージ3. Skills — Reusable Prompt Packages

3.1 Skills とは何か3.1 What are Skills?

Skill はスラッシュコマンド（/deploy、/review 等）として呼び出せる、再利用可能なプロンプト指示書である。「毎回同じ指示を書く」という非効率を排除し、チームや個人のベストプラクティスをパッケージ化する。 Skills are reusable prompt instruction packages invocable as slash commands (/deploy, /review, etc.). They eliminate the inefficiency of "writing the same instructions every time" and package best practices for teams and individuals.

Agent が「誰がやるか」を定義するなら、Skill は「何をやるか」を定義する。 If an Agent defines "who does it," a Skill defines "what to do."

3.2 Skill の定義方法3.2 How to define a Skill

Skill はディレクトリ構造で管理する。中核は SKILL.md ファイルだ。 Skills are managed as directory structures. The core is the SKILL.md file.

# ディレクトリ構造 ~/.claude/skills/deploy/ ├── SKILL.md # 必須 — フロントマター + 指示 ├── checklist.md # 任意 — 補助資料 └── scripts/ └── validate.sh # 任意 — ヘルパースクリプト# Directory structure ~/.claude/skills/deploy/ ├── SKILL.md # Required — frontmatter + instructions ├── checklist.md # Optional — supporting material └── scripts/ └── validate.sh # Optional — helper scripts
# SKILL.md の例 --- name: deploy description: 本番環境へのデプロイを実行 user-invocable: true allowed-tools: Bash(npm *), Read context: fork agent: general-purpose argument-hint: [environment] --- ## デプロイ手順 1. テストスイートを実行し、全パスを確認 2. ビルドを実行 3. $ARGUMENTS 環境にデプロイ 4. ヘルスチェックを実行# SKILL.md example --- name: deploy description: Execute deployment to production user-invocable: true allowed-tools: Bash(npm *), Read context: fork agent: general-purpose argument-hint: [environment] --- ## Deployment procedure 1. Run the test suite and confirm all pass 2. Execute the build 3. Deploy to the $ARGUMENTS environment 4. Run health checks

注目すべきは context: fork フィールドだ。これを指定すると、Skill は独立した Agent コンテキストで実行される。メインの会話を汚さずに複雑な作業を遂行できる。 The context: fork field deserves attention. When specified, the Skill executes in an isolated Agent context. Complex work can be performed without polluting the main conversation.

3.3 動的コンテキスト注入3.3 Dynamic context injection

Skill の強力な機能の一つが、シェルコマンドの結果をプロンプトに動的に注入する ! バッククォート構文だ。 One of the most powerful Skill features is the ! backtick syntax, which dynamically injects shell command output into the prompt.

# PR レビュー Skill の例 --- name: pr-summary context: fork agent: Explore --- ## PR の変更内容 - Diff: !`gh pr diff` - 変更ファイル: !`gh pr diff --name-only` 上記の差分を分析し、レビューコメントを作成してください。# PR review Skill example --- name: pr-summary context: fork agent: Explore --- ## PR changes - Diff: !`gh pr diff` - Changed files: !`gh pr diff --name-only` Analyse the diff above and draft review comments.

!`gh pr diff` はClaude が処理を開始する前にシェルで実行され、その出力がプロンプト内に展開される。つまり、Skill は静的な指示書ではなく、実行時に動的にコンテキストを取得できる「生きた指示書」になる。 !`gh pr diff` is executed in the shell before Claude begins processing, and the output is expanded within the prompt. In other words, a Skill is not a static instruction sheet — it becomes a "living document" that dynamically acquires context at runtime.

Skill と Agent の関係: Skill = 「何をするか」の定義。Agent = 「誰がするか」の定義。context: fork + agent: で両者を接続すると、「この専門家にこの仕事を頼む」というパターンが成立する。 Skill-Agent relationship: Skill = definition of "what to do." Agent = definition of "who does it." Connecting both via context: fork + agent: creates the pattern: "assign this job to this specialist."

4. Hooks — 「止める装置」と「自動実行装置」4. Hooks — The "Stop" Mechanism and the "Auto-Execute" Mechanism

4.1 Hooks とは何か4.1 What are Hooks?

Hooks は Claude Code のライフサイクルイベントに応答して自動実行されるシェルコマンドだ。Agent や Skill が「AI の判断」に依存するのに対し、Hooks は決定論的に動作する。AI が判断を誤っても、Hook が物理的に止める。 Hooks are shell commands that execute automatically in response to Claude Code lifecycle events. While Agents and Skills depend on "AI judgement," Hooks operate deterministically. Even if the AI makes a wrong call, the Hook physically stops it.

これが重要な理由: AI は確率的に動作する。「絶対にやってはいけないこと」をプロンプトで禁止しても、100%の遵守は保証できない。Hooks は確率ではなく、コードで強制する。 This matters because AI operates probabilistically. Even if you prohibit something via prompt, 100% compliance cannot be guaranteed. Hooks enforce rules not with probability, but with code.

4.2 主要なライフサイクルイベント4.2 Key lifecycle events

イベントEvent 発火タイミングWhen it fires 用途例Use case
PreToolUse ツール実行前Before tool execution 危険なコマンドのブロック、パラメータ検証Block dangerous commands, validate parameters
PostToolUse ツール実行成功後After tool success 自動フォーマット、リンター再実行Auto-format, re-run linters
SessionStart セッション開始時Session start 環境セットアップ、コンテキスト読み込みEnvironment setup, context loading
Stop Claude の応答完了時When Claude finishes responding 最終検証、チェックリスト確認Final verification, checklist confirmation
UserPromptSubmit プロンプト送信前Before prompt processing 入力の前処理・バリデーションInput preprocessing and validation
FileChanged 監視ファイルの変更時When watched files change リアクティブな更新トリガーReactive update triggers

4.3 設定例4.3 Configuration example

Hooks は settings.json で設定する。以下は「TypeScript ファイルを編集したら自動で Prettier を実行する」例だ。 Hooks are configured in settings.json. Below is an example that "automatically runs Prettier when a TypeScript file is edited."

{ "hooks": { "PostToolUse": [ { "matcher": "Edit|Write", "if": "Edit(*.ts)", "hooks": [ { "type": "command", "command": "npx prettier --write \"$FILE_PATH\"", "timeout": 30 } ] } ] } }

Hook の戻り値も重要だ。Exit code 0 はアクション続行、Exit code 2 はアクションをブロックする。ブロック理由を stderr に書くと、Claude がそれを読んで代替手段を考える。 Hook return values are also important. Exit code 0 allows the action to proceed; Exit code 2 blocks it. If you write a reason to stderr, Claude reads it and considers alternatives.

設計原則Design principle

Hooks の量産は解決策ではない。根本対策は Skill の手順に正しいワークフローを組み込むこと。Hooks はそのバックストップ（最終防衛線）として最小限に留める。 Mass-producing Hooks is not the solution. The root fix is to embed the correct workflow within Skill procedures. Hooks serve as a backstop (last line of defence) and should be kept to a minimum.

5. MCP — 外部世界への接続プロトコル5. MCP — The Protocol for Connecting to the Outside World

5.1 MCP とは何か5.1 What is MCP?

Model Context Protocol (MCP) は、Claude Code を外部ツール・API・データベースと接続するオープンソース標準プロトコルである。MCP サーバーは「ツール」（Claude が呼び出せる関数）、「リソース」（Claude が読めるデータ）、「プロンプト」（テンプレート）を公開する。 Model Context Protocol (MCP) is an open-source standard protocol connecting Claude Code to external tools, APIs, and databases. MCP servers expose "tools" (functions Claude can call), "resources" (data Claude can read), and "prompts" (templates).

これにより、Claude Code の能力は文字通り無制限に拡張できる。GitHub でコードを検索する、PostgreSQL にクエリを投げる、Notion にページを作る、Slack にメッセージを送る — すべて MCP 経由で実現可能だ。 This means Claude Code's capabilities can be extended literally without limit. Searching code on GitHub, querying PostgreSQL, creating Notion pages, sending Slack messages — all achievable via MCP.

5.2 接続方式5.2 Connection methods

トランスポートTransport 用途Use case 例Example
stdio ローカルプロセスLocal processes npx -y @github/mcp-server
HTTP リモートサーバー（推奨）Remote servers (recommended) https://mcp.notion.com/mcp

5.3 設定例5.3 Configuration example

# プロジェクトルートの .mcp.json { "mcpServers": { "github": { "type": "stdio", "command": "npx", "args": ["-y", "@github/mcp-server"], "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" } }, "notion": { "type": "http", "url": "https://mcp.notion.com/mcp", "headers": { "Authorization": "Bearer ${NOTION_TOKEN}" } } } }# .mcp.json at project root { "mcpServers": { "github": { "type": "stdio", "command": "npx", "args": ["-y", "@github/mcp-server"], "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" } }, "notion": { "type": "http", "url": "https://mcp.notion.com/mcp", "headers": { "Authorization": "Bearer ${NOTION_TOKEN}" } } } }

MCP サーバーは Agent にスコープすることも可能だ。「データベース分析 Agent だけが PostgreSQL に接続できる」という設計により、最小権限の原則を実現できる。 MCP servers can also be scoped to specific Agents. The design principle of least privilege is achievable — for example, "only the database analyst Agent can connect to PostgreSQL."

6. 4つのレイヤーの組み合わせ方6. How to Combine the Four Layers

各コンポーネントは単独でも有用だが、真価を発揮するのは組み合わせたときだ。以下に代表的なパターンを示す。 Each component is useful on its own, but the real power emerges in combination. Below are representative patterns.

Pattern A: Skill + Agent（タスク委譲）Pattern A: Skill + Agent (task delegation)

/code-review Skill が context: fork で専用 Agent にレビューを委譲。メインスレッドはレビュー結果だけを受け取り、コンテキストを消費しない。 The /code-review Skill delegates review to a dedicated Agent via context: fork. The main thread receives only the results, consuming no context.

Pattern B: Hook + Skill（自動品質保証）Pattern B: Hook + Skill (automated quality assurance)

PostToolUse Hook がファイル編集を検知し、自動で Prettier を実行。さらに Stop Hook でリンターを走らせ、問題があれば Claude にフィードバック。Skill 側にもワークフローを記述し、Hook は最終防衛線として機能。 A PostToolUse Hook detects file edits and auto-runs Prettier. A Stop Hook then runs the linter and feeds back any issues to Claude. The Skill also describes the workflow; the Hook serves as the last line of defence.

Pattern C: Agent + MCP（スコープ付き外部接続）Pattern C: Agent + MCP (scoped external access)

データ分析 Agent に PostgreSQL の MCP サーバーのみを付与。この Agent はデータベースにクエリできるが、ファイル編集やGitHub操作はできない。最小権限の原則がアーキテクチャレベルで実現される。 Grant only the PostgreSQL MCP server to a data analysis Agent. This Agent can query the database but cannot edit files or perform GitHub operations. The principle of least privilege is achieved at the architectural level.

Pattern D: 並列 Agent オーケストレーションPattern D: Parallel Agent orchestration

メインスレッドが3つの Agent を並列起動: セキュリティレビュー、パフォーマンス分析、テスト実行。それぞれ独立したコンテキストで動作し、結果がメインスレッドに集約される。直列実行の3倍以上の速度で、かつメインのコンテキストを汚さない。 The main thread launches three Agents in parallel: security review, performance analysis, and test execution. Each operates in an isolated context, and results are aggregated by the main thread. More than three times faster than sequential execution, without polluting the main context.

設計の指針: 「何を」= Skill、「誰が」= Agent、「強制」= Hook、「外部接続」= MCP。この4軸で自分のワークフローを分解すれば、最適な構成が見えてくる。 Design guideline: "What" = Skill, "Who" = Agent, "Enforce" = Hook, "External" = MCP. Decompose your workflow along these four axes and the optimal configuration will emerge.

7. Plugins とエコシステムの現在地7. Plugins and the Current State of the Ecosystem

Skills、Agents、Hooks、MCP サーバーの定義を一つにパッケージ化し、配布する仕組みが Plugins だ。 Plugins are the mechanism for packaging and distributing Skills, Agents, Hooks, and MCP server definitions together.

# Plugin のディレクトリ構造 my-plugin/ ├── plugin.json # メタデータ ├── skills/ # Skill 定義群 │ └── deploy/ │ └── SKILL.md ├── agents/ # Agent 定義群 │ └── reviewer.md ├── hooks/ # Hook 設定 │ └── hooks.json └── .mcp.json # MCP サーバー定義# Plugin directory structure my-plugin/ ├── plugin.json # Metadata ├── skills/ # Skill definitions │ └── deploy/ │ └── SKILL.md ├── agents/ # Agent definitions │ └── reviewer.md ├── hooks/ # Hook configuration │ └── hooks.json └── .mcp.json # MCP server definitions

配布は GitHub リポジトリ経由で行われ、claude plugin install コマンドでインストールする。Anthropic 公式のプラグインレジストリ（anthropics/claude-plugins-official）も GitHub 上に存在する。 Distribution occurs via GitHub repositories and plugins are installed with the claude plugin install command. Anthropic's official plugin registry (anthropics/claude-plugins-official) exists on GitHub.

エンタープライズ環境では、管理者がマネージド設定を通じて組織全体にプラグインを配布できる。これにより「全社で同じコードレビュー基準を使う」「承認されたMCPサーバーのみ許可する」といったガバナンスが可能になる。 In enterprise environments, administrators can distribute plugins organisation-wide through managed settings. This enables governance such as "use the same code review standards across the company" or "allow only approved MCP servers."

8. 結論 — 「使い方を覚える」から「環境を設計する」へ8. Conclusion — From "Learning to Use" to "Designing Your Environment"

Claude Code の拡張スタックを一文で要約するなら: The Claude Code extensibility stack, summarised in one sentence:

Skill が「何をするか」を定義し、Agent が「誰がするか」を決め、Hook が「やってはいけないこと」を強制し、MCP が「外の世界」と繋ぐ。この4層の組み合わせが、チャットボットをプログラマブルプラットフォームに変える。 Skills define "what to do," Agents decide "who does it," Hooks enforce "what must not happen," and MCP connects to "the outside world." The combination of these four layers transforms a chatbot into a programmable platform.

定義ファイルはすべて Markdown + YAML であり、プログラミング言語の知識は不要だ。つまり、ファイナンスのプロフェッショナルが、自分の業務ワークフローを AI 環境に直接組み込むことが技術的に可能な時代になった。 All definition files are Markdown + YAML, requiring no programming language knowledge. This means we have entered an era where finance professionals can technically embed their business workflows directly into an AI environment.

重要なのは「上手くプロンプトを書く」ことではない。自分の仕事を Agents / Skills / Hooks / MCP の組み合わせとして設計することだ。一度設計した環境は蓄積し、複利で生産性を向上させる。 What matters is not "writing better prompts." It is designing your work as a composition of Agents, Skills, Hooks, and MCP. Once designed, the environment accumulates and improves productivity with compound returns.

次回は、このアーキテクチャを Finance Professional の実務（DD資料作成、バリュエーション、監査対応）にどう適用するか、具体的なケーススタディで解説する。 In the next instalment, we will explore how to apply this architecture to Finance Professional workflows (DD material preparation, valuation, audit support) through concrete case studies.
